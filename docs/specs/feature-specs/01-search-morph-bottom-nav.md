# 하단 검색 모핑 (Bottom Nav Search Morph)

## 1. 개요

- **해결하는 문제**: 현재 검색 진입은 두 곳에 분산되어 있다. 홈(`/`) 상단에 슬라이드로 펼쳐지는 검색 입력창이 있고, 동시에 하단 `BottomNav`의 돋보기 버튼이 그 입력창을 토글한다. 검색 UI(상단 입력)와 검색 트리거(하단 돋보기)가 화면의 양 끝에 떨어져 있어, 엄지 도달 영역(하단)에서 버튼만 누른 뒤 시선·손가락을 다시 화면 상단으로 옮겨 입력해야 한다. 모바일 단일 손 조작에서 인터랙션 동선이 끊기고, 상단 검색바가 펼쳐지는 동안 피드 상단 콘텐츠가 밀려 레이아웃이 점프한다. 또한 검색 진입점이 두 개라 "검색이 어디 있는지" 인지 비용이 발생한다.
- **제공하는 가치**:
  - 검색의 진입·입력·결과 확인이 모두 화면 하단 한 곳(엄지 영역)에서 끝나 단일 손 조작 동선이 끊기지 않는다. 진입점이 1개로 일원화되어 검색 위치에 대한 인지 비용을 제거한다.
  - 상단 검색바 슬라이드가 제거되어 피드 상단 레이아웃 점프(layout shift)가 사라진다. 검색 모드 진입 시 푸터 컨테이너 안에서만 형태가 바뀌므로 본문 스크롤 위치가 보존된다.
  - 기존 `glass-bar rounded-full` 알약형 푸터의 시각적 연속성을 유지한 채 모핑 전환으로 "같은 객체가 변형된다"는 직관적 메탈모델을 제공한다.
- **범위**:
  - **다루는 것**:
    - `BottomNav` 컨테이너 내부에서 4버튼(돋보기·좋아요·공유·사람) ↔ 검색 입력창 모핑 전환.
    - 돋보기 탭 → 검색 모드 진입, 외부 탭/포커스 아웃 → 4버튼 모드 복귀.
    - 홈 상단 슬라이드 검색 입력창(`page.tsx`)의 제거 및 검색 UI의 하단 일원화.
    - 비(非)홈 페이지에서 돋보기 탭 시 홈 이동 후 검색 모드 진입(기존 동작 확장).
    - `SearchContext`(클라이언트 측 피드 필터)의 그대로 재사용.
  - **다루지 않는 것**:
    - 검색 알고리즘/스코어링 변경 또는 신규 검색 API (현행 클라이언트 includes 필터 유지 — `SearchContext` / `page.tsx` `filtered` 로직).
    - 검색 자동완성·최근 검색어·추천어 (본 명세 범위 외, 향후 별도 기능).
    - 좋아요·공유 버튼의 실제 기능 구현 (각 버튼은 현재 `준비 중` placeholder 상태 유지).
    - 로고/브랜드 마크 교체 (기능 04).
    - 스켈레톤/로딩 표준 적용 (기능 03 — 본 기능의 검색 결과 Loading 상태는 홈 피드 기존 로딩을 따른다).

## 2. 사용자 스토리 + 인수 조건

### US-1: 하단에서 검색 모드로 모핑 진입
As a **레시피를 찾는 사용자**, I want **하단 푸터의 돋보기를 탭하면 그 자리에서 검색 입력창이 나타나길**, so that **손가락을 화면 상단으로 옮기지 않고 검색을 시작한다**

**인수 조건:**
- [ ] AC-1-1: 홈에서 돋보기 버튼을 탭하면 4버튼(돋보기·좋아요·공유·사람)이 사라지고 동일한 `glass-bar rounded-full` 컨테이너 안이 검색 입력창으로 전환된다. 컨테이너의 둥근 알약형 테두리·위치·폭(`w-[min(92vw,420px)]`)은 전환 전후 동일하게 유지된다(컨테이너 자체는 언마운트/리마운트되지 않는다).
- [ ] AC-1-2: 버튼→입력창 전환은 200~280ms 사이의 모핑 애니메이션(버튼 묶음 fade-out + 입력창 fade-in/scale-in)으로 처리되며, `prefers-reduced-motion: reduce` 환경에서는 애니메이션 없이 즉시 전환된다.
- [ ] AC-1-3: 검색 모드로 진입한 직후(전환 애니메이션 종료 시점 기준 ≤ 350ms) 입력창에 자동 포커스가 걸려 모바일에서 키보드가 즉시 올라온다.
- [ ] AC-1-4: 검색 모드일 때 돋보기 버튼의 논리 상태(`SearchContext.open === true`)가 켜지며, 입력창에는 `aria-label="레시피, 재료, 셰프 검색"` 및 placeholder가 표시된다.

### US-2: 검색 입력으로 피드를 실시간 필터링
As a **사용자**, I want **하단 입력창에 키워드를 입력하면 홈 피드가 즉시 좁혀지길**, so that **원하는 레시피를 빠르게 찾는다**

**인수 조건:**
- [ ] AC-2-1: 입력값은 `SearchContext.query`에 반영되고, 홈 피드는 기존 클라이언트 필터(`title`·`user.nickname`·`description`을 공백 결합 후 소문자 `includes` 매칭) 기준으로 필터링된다. 신규 API 호출은 발생하지 않는다.
- [ ] AC-2-2: 입력 후 피드 반영은 키 입력당 16ms(1프레임) 이내의 클라이언트 리렌더로 처리되며 네트워크 왕복이 없다(현행 `useMemo` 필터 동작 보존).
- [ ] AC-2-3: 검색어가 비어 있지 않은데 매칭 결과가 0건이면 피드 영역에 "검색 결과가 없어요" Empty 상태가 표시되고, 검색어가 비면 전체 피드로 복귀한다.
- [ ] AC-2-4: 검색 모드에서도 입력창 좌측에 돋보기 아이콘(시각적 어포던스)이 표시되고, 입력값이 있을 때 우측에 지우기(clear, ✕) 버튼이 나타나 탭 시 `query`를 비우되 검색 모드는 유지한다.

### US-3: 외부 탭/포커스 아웃 시 버튼 모드 복귀
As a **사용자**, I want **검색 입력창 바깥을 탭하면 다시 원래 4버튼 푸터로 돌아가길**, so that **검색을 끝낸 뒤 일반 내비게이션으로 자연스럽게 복귀한다**

**인수 조건:**
- [ ] AC-3-1: 검색 입력창/푸터 컨테이너 바깥 영역(피드, 헤더, 빈 공간 등)을 포인터로 탭하거나 입력창이 포커스를 잃으면, 입력창이 4버튼 묶음으로 역방향 모핑되어 복귀한다.
- [ ] AC-3-2: 버튼 모드로 복귀하면 `SearchContext.open`이 `false`가 되고, 기존 동작에 따라 검색어(`query`)는 초기화되어 피드가 전체 목록으로 돌아간다(현행 `page.tsx`의 "slides away → clear query" 규칙 유지).
- [ ] AC-3-3: 키보드 사용자가 입력창에서 `Esc` 키를 누르면 검색 모드가 닫히고 4버튼 모드로 복귀하며, 복귀 후 포커스는 돋보기 버튼으로 돌아간다.
- [ ] AC-3-4: 입력창 내부(아이콘·clear 버튼 포함)를 탭하는 동작은 "외부 탭"으로 간주되지 않아 모드가 닫히지 않는다.

### US-4: 비(非)홈 페이지에서의 검색 진입
As a **레시피 상세/프로필 등 다른 페이지에 있는 사용자**, I want **하단 돋보기를 탭하면 홈으로 가서 바로 검색 상태가 되길**, so that **어느 화면에서든 한 번의 탭으로 검색을 시작한다**

**인수 조건:**
- [ ] AC-4-1: 현재 경로가 `/`가 아닐 때 돋보기를 탭하면 `/`로 라우팅된 뒤 검색 모드가 자동 진입(`SearchContext.open = true`)되며, 홈 도착 후 입력창에 포커스가 걸린다(기존 `onSearch`의 `router.push("/") + setSearchOpen(true)` 패턴 확장).
- [ ] AC-4-2: 홈으로 이동하는 동안에도 푸터 컨테이너는 유지되며(전역 fixed 컴포넌트), 라우팅 완료 후 입력창 형태로 표시된다.
- [ ] AC-4-3: 비홈 페이지에서 좋아요·공유·사람 버튼의 기존 동작(placeholder / 프로필 링크)은 변경되지 않는다.

## 3. UI/UX 명세

### 컴포넌트 계층

```
RootLayout
├── Header (검색 입력 제거 영향 없음 — 헤더엔 원래 검색 없음)
├── main
│   └── Home (page.tsx)
│       ├── (제거됨) 상단 슬라이드 검색 입력창
│       └── FeedList
│           ├── FeedPost ...
│           └── EmptyState (검색 결과 없음 / 레시피 없음 / 에러)
└── BottomNav (fixed, glass-bar rounded-full 컨테이너)  ← 모핑 주체
    └── MorphContainer  (동일 컨테이너, 모드에 따라 내용 교체)
        ├── [버튼 모드] NavButtons
        │   ├── Button(돋보기, onClick → 검색 모드 진입)
        │   ├── Button(좋아요, 준비 중)
        │   ├── Button(공유, 준비 중)
        │   └── Link(사람 → 프로필/로그인)
        └── [검색 모드] SearchField
            ├── Search 아이콘 (좌)
            ├── input[type=search] (query 바인딩)
            └── ClearButton(✕, query 있을 때, 우)
```

### 5가지 화면 상태

본 기능은 "검색 결과"를 화면 상태로 매핑한다. 검색은 클라이언트 필터이므로 네트워크 Loading/Error는 홈 피드의 초기 로드 상태에 종속된다.

| 상태 | 조건 | UI 표현 | 사용자 액션 |
|------|------|---------|------------|
| Empty | 검색 모드 진입했으나 입력 전(`query` 빈 문자열), 피드는 전체 노출 | 푸터=입력창(placeholder "레시피, 재료, 셰프 검색"), 피드=전체 레시피 목록 / 레시피 자체가 0건이면 "아직 레시피가 없어요"(🍳) | 키워드 입력, 외부 탭/Esc로 닫기 |
| Loading | 홈 피드 초기 fetch 진행 중(`/api/recipes` 응답 대기). 검색 자체엔 로딩 없음 | 피드 영역에 기존 `animate-pulse` 스켈레톤 2장(기능 03 적용 시 표준 스켈레톤+원형 프로그레스로 교체). 푸터는 입력 가능 상태 유지 | 입력은 가능하나 결과는 데이터 도착 후 필터링됨 |
| Partial | 피드 첫 페이지(30건)만 로드되고 `nextCursor` 존재. 검색은 로드된 항목 한정 필터 | 검색 중에는 "더 보기" 버튼 숨김(`!searching && nextCursor`), 현재까지 로드된 항목 내에서만 결과 표시. 결과 하단에 추가 로딩 없음 | 검색어를 비우면 "더 보기"로 추가 로드 가능 |
| Error | 홈 피드 초기 fetch 실패(`error === true`) | 피드 영역에 "레시피를 불러오지 못했어요"(🥕) Empty/Error 카드. 푸터=입력창은 정상 표시되나 필터 대상 데이터가 없음 | 페이지 새로고침으로 재시도, 외부 탭으로 검색 모드 닫기 |
| Ideal | 데이터 정상 + 검색어 입력 + 1건 이상 매칭 | 푸터=입력창(좌 돋보기, 우 clear 버튼), 피드=매칭된 `FeedPost` 목록만 노출. 결과 0건이면 "검색 결과가 없어요"(🔍, 검색어 echo) | 검색어 수정/삭제, clear 탭, 결과 카드 진입, 외부 탭/Esc 닫기 |

> 5가지 상태 모두 필수. 본 기능에서 "검색 결과 없음"은 Ideal 상태의 0건 분기(🔍)로, "입력 전"은 Empty로 분리해 매핑한다.

### 반응형 대응

- **Mobile (< 768px)**: 기본 타깃. 푸터 폭 `w-[min(92vw,420px)]`. 검색 모드 진입 시 입력창이 컨테이너 가용 폭을 모두 차지(좌 아이콘 + input + 우 clear). 포커스 시 OS 키보드가 올라오며, 푸터는 `fixed bottom-5`로 키보드 위에 머무르지 않을 수 있으므로(브라우저별 차이) 입력은 키보드 바로 위 시야에 들어오도록 보장하지 않아도 되나, 진입 시 자동 포커스로 입력 흐름을 끊지 않는다.
- **Tablet (768px ~ 1024px)**: 폭 상한 420px 유지(중앙 정렬). 버튼/입력창 크기는 모바일과 동일. 터치·마우스 양쪽 외부 탭 감지 동작.
- **Desktop (> 1024px)**: 동일 컴포넌트를 마우스 환경에서 사용. 외부 클릭으로 닫힘, hover 상태(돋보기/사람 버튼 `hover:text-ink-soft`) 유지. 입력창 폭 상한 420px 그대로(앱은 모바일 우선 레이아웃 `max-w-[520px]`).

### 인터랙션 상세

| 인터랙션 | 동작 |
|----------|------|
| 돋보기 탭 (홈) | `toggleSearch()` 호출(현행 유지). `open: false→true`. 버튼 묶음 fade-out(120ms) 후 입력창 fade-in + 미세 scale(0.96→1, 160ms)로 모핑. 전환 완료 후 input `focus()`. |
| 돋보기 탭 (비홈) | `router.push("/")` 후 `setSearchOpen(true)`(현행 유지). 홈 마운트 시점에 `open===true`이면 입력창 모드로 렌더 + 포커스. |
| 키 입력 | `onChange`로 `setQuery(value)`. 디바운스 없음(클라이언트 필터가 가벼움). `useMemo` 재계산으로 피드 즉시 갱신. |
| clear(✕) 탭 | `setQuery("")`. 검색 모드 유지, 입력창 재포커스. 피드는 전체로 복귀. |
| 외부 탭(포인터) | 컨테이너 ref 밖 `pointerdown` 감지 시 `setOpen(false)`. 입력창 fade-out → 버튼 묶음 fade-in(역방향 모핑). `query`는 `open:false` effect로 초기화. |
| 포커스 아웃(blur) | 입력창이 포커스를 잃고 새 포커스 대상이 컨테이너 밖이면 닫기. 컨테이너 내부(clear 버튼 등) 이동은 무시. |
| Esc 키 | 입력창에서 `Escape` keydown 시 `setOpen(false)` + 돋보기 버튼으로 포커스 환원. |
| reduced-motion | `prefers-reduced-motion: reduce`면 fade/scale 트랜지션 제거, 즉시 토글. |

### 접근성 (a11y)

- 모핑 컨테이너의 모드 전환은 `aria-expanded`(돋보기 버튼) 또는 검색 영역의 `role="search"`로 의미 전달. 검색 모드 진입 시 입력창을 `role="search"` 래퍼로 감싸고 input에 `aria-label` 부여.
- 버튼 모드의 돋보기 버튼은 기존 `aria-label="검색"` + `aria-pressed`/`aria-expanded`로 모드 상태를 보조기술에 전달.
- 키보드 단독 사용자: Tab으로 입력창/clear 진입, Esc로 닫기·포커스 환원(AC-3-3), 외부로 Tab 이동 시 자연 blur 닫힘.
- clear 버튼은 `aria-label="검색어 지우기"`. 입력창 자동 포커스는 사용자가 명시적으로 검색을 트리거한 경우(돋보기 탭)에만 발생해 포커스 탈취로 인지되지 않도록 한다.
- `nav`의 `aria-label="하단 내비게이션"`은 유지하되, 검색 모드에서는 내부 `role="search"`가 검색 영역임을 별도로 표현.

### 시각 디자인 스펙 (해당 시)

신규 팔레트 없음 — 기존 흑백 라인 토큰 재사용. 컨테이너=`glass-bar rounded-full`, 입력=`input-field rounded-full` 변형(푸터 내부에 맞춰 `border`/`box-shadow`는 컨테이너가 이미 제공하므로 input은 투명 배경 + 무테 가능), 아이콘=`Search`(현 `text-ink-faint`), 활성 텍스트=`text-ink`.

| 요소 | 배경 | 텍스트/아이콘 | 비고 |
|------|------|----------------|------|
| 푸터 컨테이너(양 모드 공통) | `glass-bar` (rgba(255,255,255,0.88)+blur) | — | `rounded-full`, 폭/위치 고정 |
| 버튼 모드 아이콘(비활성) | 투명 | `--color-ink-faint` (#9a9a9a) | hover 시 `--color-ink-soft` |
| 검색 모드 입력 텍스트 | 투명(컨테이너 위) | `--color-ink` (#141414) | placeholder `--color-ink-faint` |
| clear 버튼 | 투명 | `--color-ink-faint` → hover `--color-ink-soft` | rounded-full 탭 영역 |

> 픽셀 단위 세부 스펙은 디자인 SSOT(빌드 단계 `frontend-design`)가 관장. 본 명세는 동작·토큰 재사용 원칙만 고정한다.

**변경 없음 — 클라이언트 측 필터, 기존 `SearchContext` 재사용. 신규 API 없음.**

## 6. 비즈니스 로직

### 핵심 로직 상세 — 모핑/검색 상태 전이

푸터는 단일 불리언 `SearchContext.open`을 모드 스위치로 사용한다. `open`이 모핑 방향과 피드 필터 노출을 동시에 결정한다.

상태 전이:

```
[버튼 모드]  ──돋보기 탭(홈)──▶  [검색 모드]
[버튼 모드]  ──돋보기 탭(비홈): push("/") + setOpen(true)──▶  [검색 모드(홈에서)]
[검색 모드]  ──외부 탭 / blur(컨테이너 밖) / Esc──▶  [버튼 모드]
[검색 모드]  ──clear(✕)──▶  [검색 모드]  (query만 ""로, 모드 유지)
```

| 조건 | 결과 |
|------|------|
| `open === false` | 푸터 = 4버튼 묶음 렌더, `query`는 effect로 `""` 강제(닫힘 시 초기화) |
| `open === true` && 경로 `/` | 푸터 = 입력창 렌더 + 자동 포커스, 피드 = `filtered` 적용 |
| `open === true` && 경로 ≠ `/` | (도달 직전 단계) `router.push("/")` 트리거됨 → 홈 마운트 후 위 행 적용 |
| `query.trim().length > 0` | 피드 = 필터 결과, "더 보기" 숨김(`searching` true) |
| `query.trim().length === 0` && `open===true` | 피드 = 전체, 입력창 placeholder 표시 |

검색 매칭 알고리즘(현행 보존, `page.tsx` `filtered`):
1. `q = query.trim().toLowerCase()`. 빈 문자열이면 전체 반환.
2. 각 레시피에 대해 `[title, user.nickname, description ?? ""].join(" ").toLowerCase().includes(q)`.
3. 매칭된 항목만 `FeedPost`로 렌더.

### 유효성 검사

| 필드 | 규칙 | 에러 메시지 |
|------|------|------------|
| `query`(검색어) | 자유 텍스트, 제약 없음. 앞뒤 공백은 매칭 시 `trim()`으로 무시 | (없음 — 검증 실패 개념 없음) |
| `query` 0건 매칭 | 입력은 유효, 결과만 없음 | "검색 결과가 없어요 / ‘{query}’에 맞는 레시피를 찾지 못했어요." (Empty UI) |

> 검색어 자체에는 형식 검증/길이 제한이 없다(클라이언트 필터, 인젝션 표면 없음). 매우 긴 입력도 `includes`로 안전.

### 엣지케이스

- **케이스 1 — 모드 전환 도중 빠른 재탭**: 모핑 애니메이션 진행 중 돋보기/외부를 빠르게 재탭하면 `open` 토글이 연속 발생할 수 있다 → `open` 단일 소스 오브 트루스를 사용하고, 트랜지션은 CSS가 현재 `open` 값으로 자연 보간하도록 해 중간 상태가 어긋나지 않게 한다(애니메이션 큐잉/락 불필요).
- **케이스 2 — 비홈에서 돋보기 탭 후 라우팅 지연**: `push("/")` 직후 `setOpen(true)`했으나 홈 마운트가 늦으면, 홈 `page.tsx`의 마운트 effect가 `open===true`를 읽어 포커스/필터를 적용한다(현행 `useEffect([searchOpen])` 보존). 라우팅 중 푸터는 fixed 전역 컴포넌트라 끊기지 않는다.
- **케이스 3 — 검색 모드에서 결과 카드로 이동**: 검색 중 `FeedPost`를 탭해 상세로 라우팅하면 경로가 `/` 이탈 → 푸터는 버튼 모드 기준으로 동작(상세에서 `open`은 다음 진입까지 의미 없음). 뒤로 가기로 홈 복귀 시 `open` 상태/`query`는 닫힘 규칙에 따라 초기화되어 전체 피드로 시작.
- **케이스 4 — 외부 탭 감지와 입력 내부 탭 충돌**: clear 버튼·아이콘 등 컨테이너 내부 탭이 "외부 탭"으로 오인되면 모드가 닫힌다 → 외부 탭 감지는 컨테이너 `ref.contains(target)`로 판정하고, `pointerdown` 단계에서 내부면 무시(AC-3-4).
- **케이스 5 — 데이터 로딩 중 검색**: 피드 fetch 완료 전 검색어를 입력하면 필터 대상이 비어 결과 0건처럼 보일 수 있다 → 로딩 중에는 피드 영역이 스켈레톤이며, 데이터 도착 후 현재 `query`로 자동 재필터(`useMemo` 의존성에 `recipes` 포함)된다.
- **케이스 6 — reduced-motion 사용자**: 모핑 트랜지션을 제거하되 모드 전환 자체와 포커스 이동은 동일하게 동작(AC-1-2).

## 7. 상태 관리

### 클라이언트 상태

검색·모핑의 단일 소스 오브 트루스는 기존 `SearchContext`다(신규 컨텍스트/스토어 없음). 모핑은 `open` 파생.

```typescript
// 기존 SearchContext (재사용, 시그니처 변경 없음)
interface SearchContextValue {
  open: boolean;          // 모핑 모드 스위치: false=버튼, true=입력창
  query: string;          // 검색어 (클라이언트 필터 키)
  setQuery: (q: string) => void;
  setOpen: (o: boolean) => void;
  toggle: () => void;     // 돋보기 탭(홈)에서 사용
}

// BottomNav 내부 로컬 상태 (모핑/외부탭 처리 전용)
interface BottomNavMorphLocal {
  containerRef: React.RefObject<HTMLDivElement>; // 외부 탭 판정 기준
  inputRef: React.RefObject<HTMLInputElement>;   // 진입 시 자동 포커스
  // 파생값(별도 state 불필요):
  mode: "buttons" | "search";                    // = open ? "search" : "buttons"
}
```

모핑 상태 머신(개념):

```
open=false  ─(toggle/setOpen true)─▶  open=true
open=true   ─(setOpen false: 외부탭·blur·Esc)─▶  open=false
            └─(query change: setQuery)──────────┘  (open 유지)
```

- `mode`는 `open`에서 파생하므로 별도 보관하지 않는다(상태 이중화 방지).
- 외부 탭/Esc/blur 핸들러는 `BottomNav`에 추가하며 모두 `setOpen(false)`로 수렴한다.
- 닫힘 시 `query` 초기화는 홈(`page.tsx`)의 기존 effect가 담당한다(`open===false → setQuery("")`). 단일 책임 유지를 위해 초기화 위치를 홈에 그대로 둔다.

### 서버 상태 (React Query / SWR)

**해당 없음 — 검색은 서버 호출이 없다.** 피드 데이터는 `page.tsx`의 기존 `fetch` + `useState`로 로드되며 본 기능은 그 로직을 변경하지 않는다(델타 계획의 별도 React Query 도입은 본 기능 범위 외).

### 낙관적 업데이트

**해당 없음 — 읽기 전용 필터 기능.** 검색은 기존 클라이언트 배열을 필터링할 뿐 쓰기 작업이 없으므로 낙관적 업데이트·롤백 개념이 존재하지 않는다.

## 8. 에러 처리

검색 자체는 네트워크/서버를 사용하지 않으므로 고유 에러가 없다. 아래는 검색이 의존하는 홈 피드 로드의 에러가 검색 UX에 미치는 영향과 복구 전략이다.

| 시나리오 | 에러코드 | 사용자 메시지 | 복구 전략 |
|----------|----------|--------------|----------|
| 피드 초기 로드 실패(검색 대상 데이터 없음) | NETWORK_ERROR / 5xx | "레시피를 불러오지 못했어요 — 일시적인 오류일 수 있어요. 잠시 후 다시 시도해주세요." (🥕) | 페이지 새로고침으로 재시도. 검색 모드는 외부 탭/Esc로 닫고 다시 진입 가능. 푸터 모핑은 정상 동작(데이터와 독립). |
| 검색 결과 0건 | (에러 아님) | "검색 결과가 없어요 — ‘{query}’에 맞는 레시피를 찾지 못했어요." (🔍) | 검색어 수정 또는 clear(✕)로 전체 피드 복귀. |
| 비홈→홈 라우팅 실패/지연 | (라우팅 예외) | (별도 에러 토스트 없음 — 라우팅은 Next 기본 처리) | 라우팅 완료 후 홈 effect가 `open` 상태를 읽어 정상 진입. 실패 시 사용자가 홈 탭으로 직접 이동 가능. |
| 자동 포커스 차단(브라우저 정책) | (소프트 실패) | (없음 — 입력창은 표시됨) | 사용자가 입력창을 직접 탭해 포커스. 모드 전환은 정상. |

> 복구 전략 핵심: 모핑/검색 UI는 데이터 로드 상태와 **독립적으로** 동작하므로, 데이터 에러 상황에서도 사용자는 검색 모드를 열고 닫을 수 있다. 데이터 복구는 새로고침 한 번으로 수렴한다.

## 14. 복잡도 + MVP 범위

- **복잡도**: low ~ medium — 신규 DB/API 없음, 기존 컨텍스트 재사용. 난이도는 (a) 컨테이너 동일성을 유지한 양방향 모핑 애니메이션, (b) 외부 탭/blur/Esc 닫힘 감지, (c) 비홈 라우팅 후 포커스 타이밍 처리에 집중된다. 데이터/권한 복잡성은 없다.

### MVP 포함 항목
- 돋보기 탭 → 입력창 모핑 진입 + 자동 포커스.
- 외부 탭 / blur / Esc 닫힘 + query 초기화.
- 기존 `SearchContext` 필터 재사용, 홈 상단 검색바 제거.
- 비홈 돋보기 → 홈 이동 후 검색 진입(기존 동작 확장).
- reduced-motion 대응, clear(✕) 버튼.

### Full 전용 항목
- 검색 모드 전환 마이크로 모션 정교화(버튼 묶음 stagger fade, 아이콘 좌측 슬라이드 등) — MVP는 단순 fade/scale로 충분.
- 검색 결과 하이라이트(매칭 텍스트 강조) — 본 기능 범위 외, 향후 검토.
- 최근 검색어/추천어 — 별도 기능으로 분리(본 명세 §1 제외 항목).
