# 기능 02: AI 이미지 생성

> 새 레시피 작성 시 사진을 직접 업로드하지 않고, 텍스트 프롬프트로 **AI가 이미지를 생성**해 레시피 사진으로 등록한다.
> 기존 수동 업로드(`ImageUploader` + `POST /api/upload`)와 **공존**하며, 생성된 이미지도 기존 업로드 자산과 동일하게 Supabase Storage(`recipe-images`)에 저장해 공개 URL로 다룬다.
> 시각 시안(5개)은 빌드 단계 frontend-design 산출물이며, 본 명세는 **동작·데이터·상태·권한·에러**만 확정한다(공유 §3.3).

## 1. 개요

- **해결하는 문제**: 레시피를 등록하려는 사용자가 적절한 음식 사진을 직접 촬영·확보하지 못해 사진 없이 등록하거나(피드 노출 품질 저하), 외부에서 무단 이미지를 가져와 저작권 위험이 발생한다. 현재 사진 영역은 카메라/갤러리 업로드만 지원하므로, 사진이 없는 사용자는 등록을 포기하거나 빈 카드로 게시한다. 피드 카드의 시각 일관성이 떨어지고, 사진 없는 레시피의 좋아요·조회율이 낮다.
- **제공하는 가치**:
  - **즉시 사진 확보**: 촬영·검색·저작권 검토 없이 프롬프트 한 줄로 음식 이미지를 생성해 사진 없는 레시피 게시를 방지(사진 미첨부 등록 비율 감소 목표).
  - **기존 흐름 무손상**: 수동 업로드와 동일한 데이터 경로(Supabase Storage → 공개 URL → `imageUrls` 배열)를 사용하므로, 다운스트림(상세/피드/수정) 로직 변경이 0.
  - **생성형 UI 경험**: 후보 이미지를 점진적으로 드러내고(generative UI) 사용자가 마음에 드는 1장을 선택하게 해, "생성→선택→등록"의 명확한 통제권을 제공.
- **범위**:
  - **다루는 것**:
    - 사진 영역 내 "AI 이미지 생성" 진입점(기존 "사진 추가" 타일과 공존하는 별도 타일/버튼).
    - 배경 블러 오버레이 + 프롬프트 입력 모달.
    - 신규 `POST /api/ai/generate-image`(Vercel AI Gateway 경유 Gemini 이미지 생성 → Supabase Storage 저장 → 공개 URL 반환).
    - 후보 이미지 점진 표시, 1장 선택 → 기존 `imageUrls`에 추가(최대 3장 제한 존중).
  - **다루지 않는 것**:
    - 수동 파일 업로드 동작(기존 `ImageUploader`/`/api/upload` — 변경 없음, 공존만).
    - 생성 이미지 편집/리터칭·인페인팅(범위 외).
    - URL/영상 분석으로 레시피 폼 자동 채움(기능 05 — `UrlRecipeImport`).
    - 로딩 UI 공용 컴포넌트 정의(기능 03이 정의, 본 기능은 `Spinner`/`Skeleton` 소비만).
    - 레시피 저장 자체(기존 `RecipeForm` 제출 흐름 — 변경 없음).

## 2. 사용자 스토리 + 인수 조건

### US-1: 프롬프트로 이미지 생성
As a **로그인한 레시피 작성자**, I want **사진 영역에서 "AI 이미지 생성"을 눌러 텍스트로 원하는 이미지를 묘사**, so that **직접 사진을 찍거나 찾지 않고도 레시피 사진을 확보한다**

**인수 조건:**
- [ ] AC-1-1: 사진 영역에 "사진 추가" 타일과 **별도로** "AI 이미지 생성" 진입점이 렌더되며, `imageUrls.length < maxImages(3)`일 때만 노출된다(3장 도달 시 진입점·업로드 타일 모두 숨김).
- [ ] AC-1-2: 진입점 클릭 시 배경(`RecipeForm` 영역)에 **블러 오버레이**가 깔리고, 프롬프트 입력창("어떤 이미지가 나왔으면 좋겠는지 입력")과 생성 버튼을 가진 모달이 뜬다.
- [ ] AC-1-3: 프롬프트가 공백이면 생성 버튼이 비활성(disabled)이며, 1자 이상 입력 시에만 활성화된다.
- [ ] AC-1-4: 생성 버튼 클릭 시 `POST /api/ai/generate-image`가 1회 호출되고, 호출 중에는 중복 제출이 차단된다(버튼 disabled + 상태 `generating`).

### US-2: 후보 점진 표시 후 선택
As a **레시피 작성자**, I want **생성된 후보 이미지를 보고 마음에 드는 1장을 선택**, so that **내 레시피에 가장 어울리는 사진만 등록한다**

**인수 조건:**
- [ ] AC-2-1: 생성 중에는 요청한 후보 수(n)만큼의 **스켈레톤 타일 + 원형 프로그레스(circular progress)** 가 표시되어 결과 레이아웃을 미리 점유한다(공유 §3.2).
- [ ] AC-2-2: 후보 이미지가 도착하는 대로 해당 타일이 스켈레톤에서 실제 이미지로 **점진 교체**된다(전부 도착 전이라도 도착분부터 선택 가능 — Partial 상태).
- [ ] AC-2-3: 후보 타일 클릭 시 선택 상태(선택 인덱스)가 갱신되고, 선택된 타일에 흑백 라인 강조(ink 테두리)가 표시된다. 한 번에 1장만 선택된다.
- [ ] AC-2-4: "이 사진 사용" 확정 시 선택된 이미지 URL이 기존 `imageUrls` 배열 끝에 추가되고, 모달이 닫히며 블러가 해제된다.
- [ ] AC-2-5: 추가 결과 `imageUrls.length`가 `maxImages(3)`를 **초과하지 않는다**(이미 3장이면 진입 자체가 막히고, 선택 확정도 거부된다).

### US-3: 실패 복구
As a **레시피 작성자**, I want **생성이 실패하면 이유와 재시도 수단을 명확히 안내받기**, so that **모달에 갇히지 않고 다시 시도하거나 수동 업로드로 전환한다**

**인수 조건:**
- [ ] AC-3-1: API가 비-2xx를 반환하면 모달 내부에 사용자 친화 에러 메시지(서버 `error` 필드)와 "다시 시도" 버튼이 표시된다(상태 `error`).
- [ ] AC-3-2: "다시 시도"는 동일 프롬프트로 재호출하며, 프롬프트 입력값은 유지된다.
- [ ] AC-3-3: 모달 외부 영역(블러 오버레이) 클릭 또는 닫기(X)·ESC 키로 모달을 닫을 수 있고, 닫으면 `imageUrls`는 변경되지 않은 채 기존 사진 영역으로 돌아간다.
- [ ] AC-3-4: AI 키 부재/쿼터 초과/세이프티 차단은 각각 구분된 메시지로 안내되며, 모두 수동 업로드 경로는 정상 동작한다(공존 불변).

## 3. UI/UX 명세

### 컴포넌트 계층

```
RecipeForm
└── ImageUploader (기존 — 변경 최소: AI 진입점 슬롯만 추가)
    ├── 기존 이미지 썸네일들 (RecipeImage.url, 최대 3)
    ├── "사진 추가" 타일 (기존, 파일 업로드)
    ├── "AI 이미지 생성" 타일 (신규 진입점) ──▶ onClick: open AiImageGenerator
    └── AiImageGenerator (신규, 조건부 마운트)
        └── BlurOverlay (fixed, 배경 블러 + 클릭 시 닫기)
            └── GeneratorPanel (glass 카드, 중앙 모달)
                ├── PanelHeader ("AI 이미지 생성" + 닫기 X)
                ├── PromptField (input-field, "어떤 이미지가 나왔으면 좋겠는지 입력")
                ├── GenerateButton (btn-primary, prompt 없으면 disabled)
                ├── CandidateGrid
                │   └── CandidateTile × n
                │       ├── Skeleton (생성 중, 기능 03 컴포넌트)
                │       ├── Spinner (원형 프로그레스, 기능 03)
                │       └── Image (도착 시 — 선택 시 ink 테두리 강조)
                ├── ErrorBlock (조건부: 메시지 + "다시 시도")
                └── PanelFooter
                    ├── Button ("취소", variant: secondary)
```

### 5가지 화면 상태

| 상태 | 조건 | UI 표현 | 사용자 액션 |
|------|------|---------|------------|
| Empty | 모달이 막 열림, 프롬프트 미입력·생성 이력 없음 (`status=idle`, candidates=[]) | 배경 블러 + 중앙 glass 카드. 프롬프트 입력창(placeholder "어떤 이미지가 나왔으면 좋겠는지 입력")과 비활성 생성 버튼. 후보 영역은 안내 문구("프롬프트를 입력하고 생성하세요")만 표시 | 프롬프트 입력 / 모달 닫기(X·ESC·바깥 클릭) |
| Loading | 생성 호출 중, 아직 후보 0개 도착 (`status=generating`, candidates=[]) | 후보 그리드에 n개 **스켈레톤 타일 + 원형 프로그레스 스피너**(트랙 `--color-line`, 진행 `--color-ink`). 생성 버튼은 "생성 중..." disabled. 프롬프트 입력 잠금 | 대기(취소·닫기로 중단 가능) |
| Partial | 일부 후보만 도착 (`status=generating`, 0 < candidates.length < n) | 도착한 타일은 실제 이미지로 교체, 미도착 타일은 스켈레톤+스피너 유지. 도착분은 즉시 선택 가능 | 도착 후보 클릭 선택 / 계속 대기 / 닫기 |
| Error | API 비-2xx 또는 네트워크 실패 (`status=error`) | 후보 그리드 자리에 ErrorBlock: 사용자 친화 메시지 + "다시 시도" 버튼. 프롬프트 입력값 유지 | "다시 시도"(동일 프롬프트 재호출) / 닫기 / 프롬프트 수정 후 재생성 |
| Ideal | 후보 전부 도착, 선택 가능 (`status=done`, candidates.length === n) | n개 후보 모두 실제 이미지. 선택 시 해당 타일 ink 테두리 강조. "이 사진 사용" 버튼 활성화 | 후보 선택 → "이 사진 사용"(imageUrls에 추가, 모달 닫힘) / 재생성 / 닫기 |

> 5가지 상태 모두 필수. "해당 없음" 불가.

### 반응형 대응
- **Mobile (< 768px)**: 모달은 화면 하단~중앙 풀폭(좌우 16px 여백), 후보 그리드 2열. 블러 오버레이 풀스크린. 닫기 X는 우상단, 푸터 버튼은 `flex-col-reverse`(취소 위·확정 아래는 기존 RecipeForm 패턴 따름).
- **Tablet (768px ~ 1024px)**: 모달 최대폭 약 560px 중앙, 후보 그리드 2열.
- **Desktop (> 1024px)**: 모달 최대폭 약 600px 중앙, 후보 그리드 2열(n=2 기본) — n이 늘면 그리드 자동 래핑.

## 4. 데이터 모델

### 신규 스키마 불필요 — 기존 `RecipeImage` 재사용

AI로 생성·저장된 이미지는 수동 업로드와 **완전히 동일한 자산**으로 취급한다. 생성 라우트가 Supabase Storage(`recipe-images`)에 저장하고 공개 URL을 반환하면, 클라이언트는 그 URL을 기존 `imageUrls: string[]`에 추가할 뿐이다. 레시피 저장 시 기존 흐름이 `RecipeImage(url, order)` 레코드를 그대로 생성한다.

| 테이블 | 변경 | 사유 |
|--------|------|------|
| `RecipeImage`(url, order) | **변경 없음** | 생성 이미지도 `url`(공개 URL) + `order`로 동일 처리. 추가 컬럼 불필요 |
| `Recipe` | **변경 없음(MVP 권장)** | 출처가 AI인지 여부를 도메인 로직이 분기하지 않음 |

**(선택) AI 생성 추적 — MVP 제외 권장 (EXPAND-only):**
운영 분석/모더레이션 목적으로 이미지 출처 추적이 필요해지면, 아래를 **nullable 추가(EXPAND-only, 기존 컬럼 변경 금지 — 공유 §3.4)** 로 제안한다. MVP에서는 불필요.

```prisma
model RecipeImage {
  // ...기존 필드 (url, order 등) — 변경 없음...
  aiGenerated  Boolean? @default(false)  // (선택) AI 생성 여부 추적
  aiPrompt     String?                   // (선택) 생성 프롬프트 보존 (모더레이션/재현)
}
```

> 위 컬럼은 **본 MVP 범위에 포함하지 않는다**. 데이터 모델 변경 없이 출시 가능. 필요 시 후속 EXPAND 마이그레이션으로만 추가.

## 5. API 명세

### POST /api/ai/generate-image
Vercel AI Gateway 경유 Gemini 이미지 생성 모델로 프롬프트→이미지 후보를 생성하고, 각 이미지를 Supabase Storage(`recipe-images`)에 저장해 공개 URL 배열을 반환한다. **server-only**(키 클라이언트 노출 금지), **로그인 필수**(iron-session). AI 호출은 `src/lib/ai/gemini.ts`에 캡슐화(공유 §3.1).

**요청:**
```typescript
interface GenerateImageRequest {
  prompt: string;   // 사용자 입력 묘사. trim 후 1~500자. 필수
  n?: number;       // 생성할 후보 수. 기본 2, 허용 1~4 (남은 슬롯 maxImages-현재수 로 클램프 권장)
}
```

**응답 200:**
```typescript
interface GeneratedImage {
  url: string;      // Supabase Storage 공개 URL (recipe-images 버킷)
}

interface GenerateImageResponse {
  images: GeneratedImage[];   // 실제 저장 성공한 후보들 (length ≤ 요청 n)
}
```

> 구현 노트: 모델이 base64(또는 바이트)로 이미지를 반환하면, 라우트가 `/api/upload`와 동일한 방식으로 `recipe-images`에 업로드(`uuid().png`, `contentType: "image/png"`, `upsert:false`) 후 `getPublicUrl`로 URL을 얻는다. 일부만 저장 성공해도 성공분만 `images`로 반환(Partial 허용). 클라이언트는 응답 도착 순서대로 후보 그리드를 채운다(점진 표시는 클라이언트 렌더링 차원에서 처리; MVP는 단일 응답으로 일괄 반환 가능, 스트리밍은 Full 항목).

**에러 코드:**
| 코드 | 상황 | 메시지 |
|------|------|--------|
| 400 | `prompt` 누락/공백/길이 초과, `n` 범위 밖 | "올바른 프롬프트를 입력해주세요." |
| 401 | 비로그인(`session.userId` 없음) | "로그인이 필요합니다." |
| 403 | 세이프티/콘텐츠 정책 차단(모델이 안전상 거부) | "해당 내용으로는 이미지를 만들 수 없어요. 다른 묘사로 시도해주세요." |
| 429 | 게이트웨이/모델 쿼터·레이트리밋 초과 | "요청이 많아요. 잠시 후 다시 시도해주세요." |
| 500 | AI 키 부재(`AI_GATEWAY_API_KEY` 미설정)·게이트웨이 오류·Storage 업로드 실패 | "AI 이미지 생성을 처리할 수 없어요. 잠시 후 다시 시도하거나 직접 업로드해주세요." |

> 키 부재(공유 §3.1)는 진입점을 숨기지 않고 호출 시 500으로 친화 처리한다. 라우트는 catch에서 키/네트워크 원인을 서버 로그로만 남기고(`console.error`), 클라이언트엔 일반 메시지를 노출한다(기존 `/api/upload` 패턴 일치).

## 6. 비즈니스 로직

### 핵심 로직 상세

**진입점 노출 분기:**
| 조건 | 진입점/업로드 타일 | 비고 |
|------|------------------|------|
| `imageUrls.length < 3` | "사진 추가" + "AI 이미지 생성" 모두 노출 | 공존 |
| `imageUrls.length === 3` | 둘 다 숨김 | 최대 3장 제한 |
| 비로그인 | 진입점은 노출하되 호출 시 401 처리 | 기존 업로드와 동일 권한 모델 |

**생성 → 선택 → 등록 흐름(상태 전이):**
```
idle ──(생성 클릭)──▶ generating ──(전부 도착)──▶ done
  ▲                      │  │                        │
  │                  (일부 도착)                  (후보 선택 + "이 사진 사용")
  │                      ▼  └──────▶ (Partial: 선택 가능)──┘
  └───(닫기/취소)────────┴──────────────────────────────┘
                         │
                    (비-2xx/네트워크)
                         ▼
                       error ──("다시 시도")──▶ generating
```

**서버 측 처리 순서(`/api/ai/generate-image`):**
1. iron-session 확인 → 비로그인 401.
2. body 파싱·검증(`prompt` trim 1~500자, `n` 1~4) → 위반 400.
3. `src/lib/ai/gemini.ts`로 게이트웨이 호출(`google/gemini-2.5-flash-image` 계열, 텍스트→이미지).
4. 세이프티 차단 응답 → 403 매핑. 쿼터/레이트 → 429 매핑.
5. 반환된 각 이미지 바이트를 `recipe-images`에 업로드(`uuid().png`) → `getPublicUrl`.
6. 저장 성공 URL만 모아 `{ images }` 200 반환. 전부 실패 시 500.

### 유효성 검사
| 필드 | 규칙 | 에러 메시지 |
|------|------|------------|
| prompt | 필수, trim 후 1~500자 | "올바른 프롬프트를 입력해주세요." |
| n | 정수, 1~4, 미지정 시 2. `min(n, 3 - imageUrls.length)`로 클램프 권장 | (조정만, 사용자 노출 없음) |
| 선택 결과 | "이 사진 사용" 시 `imageUrls.length < 3` 재확인 | "사진은 최대 3장까지 추가할 수 있어요." |

### 엣지케이스
- **케이스 1 — 생성 중 사용자가 수동 업로드로 3장을 채움**: 모달이 열린 상태에서 백그라운드 업로드로 `imageUrls`가 3장이 되면, "이 사진 사용" 확정 시 `maxImages` 가드가 거부하고 "사진은 최대 3장까지 추가할 수 있어요." 토스트 후 모달을 닫는다(데이터 불일치 방지).
- **케이스 2 — 일부 후보만 저장 성공**: 모델은 n개 생성했으나 Storage 업로드 일부 실패 → 성공분만 `images`로 반환(Partial). 0개 성공이면 500으로 처리해 Error 상태 진입.
- **케이스 3 — 세이프티 차단**: 부적절/금지 묘사로 모델이 거부 → 403 + 전용 메시지("다른 묘사로 시도"). 프롬프트 유지해 사용자가 즉시 수정 가능.
- **케이스 4 — 생성 중 모달 닫기**: in-flight 요청 결과는 폐기(클라이언트에서 무시), `imageUrls` 불변. (선택)`AbortController`로 취소 가능.
- **케이스 5 — 동시 더블클릭 생성**: `status=generating` 동안 생성 버튼 disabled로 중복 호출 차단.

## 7. 상태 관리

### 클라이언트 상태 (`AiImageGenerator` 로컬)
```typescript
type GenStatus = "idle" | "generating" | "done" | "error";

interface AiImageGeneratorState {
  open: boolean;              // 모달 open + 배경 블러 활성 여부
  prompt: string;            // 프롬프트 입력값 (Error 시에도 유지)
  status: GenStatus;         // idle | generating | done | error
  candidates: string[];      // 도착한 후보 이미지 URL (점진 추가)
  expectedCount: number;     // 요청한 n (스켈레톤 자리 수)
  selectedIndex: number | null; // 선택된 후보 인덱스 (단일 선택, 토글)
  errorMessage: string | null;  // 서버 error 필드 표시용
}
```

진입점은 `ImageUploader`(또는 `RecipeForm`)에 optional prop으로 주입되며, 선택 확정 시 기존 `onChange(images: string[])` 콜백으로 `imageUrls`에 URL을 추가한다(기존 시그니처 보존 — 공유 §3.4).

### 서버 상태 (React Query / SWR)
```
해당 없음 — 본 기능은 명령형(POST) 단발 호출이다.
캐싱 대상 목록 데이터가 없으므로 React Query/SWR 불필요.
fetch + 로컬 status 머신으로 충분(기존 ImageUploader의 fetch 패턴 일치).
```

### 낙관적 업데이트
- **해당 없음** — 생성 결과(실제 URL)가 도착한 뒤에만 `imageUrls`에 추가한다. 추측 URL을 미리 넣지 않는다(존재하지 않는 이미지 렌더 방지).

## 8. 에러 처리

| 시나리오 | 에러코드 | 사용자 메시지 | 복구 전략 |
|----------|----------|--------------|----------|
| 네트워크 오류(fetch reject) | NETWORK_ERROR | "네트워크 상태를 확인하고 다시 시도해주세요." | `status=error`, "다시 시도" 버튼(프롬프트 유지). 수동 업로드 경로는 정상 |
| 비로그인/세션 만료 | 401 | "로그인이 필요합니다." | 로그인 안내. 작성 중 입력값은 폼에 보존(모달만 닫힘) |
| 부적절 프롬프트 차단 | 403 | "해당 내용으로는 이미지를 만들 수 없어요. 다른 묘사로 시도해주세요." | 프롬프트 유지 → 수정 후 재생성 |
| 쿼터/레이트리밋 | 429 | "요청이 많아요. 잠시 후 다시 시도해주세요." | "다시 시도"(백오프 권장). 수동 업로드 대체 안내 |
| 잘못된 입력 | 400 | "올바른 프롬프트를 입력해주세요." | 입력창 포커스, 생성 버튼 재비활성 |
| AI 키 부재/게이트웨이/Storage 실패 | 500 | "AI 이미지 생성을 처리할 수 없어요. 잠시 후 다시 시도하거나 직접 업로드해주세요." | "다시 시도" + 수동 업로드 권유(공존이 안전망). 서버는 원인 로그만 |
| 일부 후보만 도착 | (Partial, 2xx) | (에러 아님) | 도착분 즉시 선택 가능, 미도착 타일 스켈레톤 유지 |

> 모든 에러 경로에서 **수동 업로드(`/api/upload`)는 영향받지 않는다**(공유 §3.4 가산성). 사용자는 언제든 모달을 닫고 직접 업로드로 전환할 수 있다.

## 14. 복잡도 + MVP 범위

- **복잡도**: **medium~high** — 신규 외부 AI 연동(Vercel AI Gateway + Gemini 이미지), 바이트→Supabase Storage 저장 파이프라인, 생성형 UI(점진 표시·선택), 5가지 상태/에러 분기(키 부재·쿼터·세이프티) 때문에 high에 근접. 단, 데이터 스키마 변경이 없고 기존 업로드 자산 경로를 재사용하므로 통합 위험은 medium 수준으로 억제된다.

### MVP 포함 항목
- 사진 영역 내 "AI 이미지 생성" 진입점(3장 제한·공존).
- 배경 블러 + 프롬프트 모달.
- `POST /api/ai/generate-image`(로그인 필수, server-only, 게이트웨이 경유, Storage 저장, 공개 URL 반환).
- 단일 응답 일괄 반환 기반 후보 그리드(스켈레톤+원형 프로그레스 로딩) + 단일 선택 → `imageUrls` 추가.
- 키 부재/쿼터/세이프티/네트워크 에러 친화 처리 + "다시 시도" + 수동 업로드 안전망.
- 데이터 모델 변경 없음(`RecipeImage` 재사용).

### Full 전용 항목
- **진짜 스트리밍 점진 표시**: 후보를 1개씩 스트림으로 받아 도착 즉시 렌더(MVP는 단일 응답 후 클라이언트가 순차 표시). 서버 스트리밍/SSE 필요로 제외.
